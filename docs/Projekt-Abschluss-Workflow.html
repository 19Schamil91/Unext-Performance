UNEXT GMBH · Abschlussplan

Projekt sauber abschließen, ohne Korrekturkreislauf

Diese Anleitung ist ein praktischer Workflow für die letzte Projektphase. Sie hilft, Zeilenumbrüche, Überschriften, Mobile-Fehler und Launch-Risiken geordnet zu sammeln, zu priorisieren und gebündelt zu beheben.

Wichtigste Regel: Ab jetzt keine Mikro-Korrekturen mehr sofort umsetzen. Erst sammeln, dann priorisieren, dann in Blöcken lösen, dann Regression prüfen.

1. Warum du gerade Zeit verlierst

Der aktuelle Kreislauf entsteht, weil einzelne sichtbare Probleme sofort korrigiert werden: ein Zeilenumbruch hier, eine Überschrift dort, danach fällt auf einer anderen Seite ein neuer Bruch auf. Das ist normal, wenn Typografie, Content und responsive Layout gleichzeitig bearbeitet werden.

Die Lösung ist nicht schnelleres Korrigieren, sondern ein anderer Modus: erst alle relevanten Probleme erfassen, dann Muster erkennen, dann systemisch lösen. Einzelne Zeilenumbrüche sind nur dann direkt relevant, wenn sie ein Launch-Problem sind.

2. Freeze-Regeln für die Abschlussphase

  • Keine neuen Design-Ideen, solange Launch-Blocker offen sind.
  • Keine neuen Komponenten ohne vorherigen codebase_explorer.
  • Keine manuellen Satzumbrüche als Standardlösung.
  • Keine P3-Schönheitskorrekturen, bevor P0/P1 erledigt sind.
  • Jede Änderung muss einem Finding oder einer klaren Launch-Aufgabe zugeordnet sein.
  • Nach jedem Fix-Block wird geprüft, aber nicht direkt eine neue offene Designrunde gestartet.
Gute Entscheidungsfrage: Verhindert dieses Problem Kontaktaufnahme, Vertrauen, Lesbarkeit, Mobile-Nutzung, SEO oder rechtliche Launch-Reife? Wenn nein, ist es wahrscheinlich P2 oder P3.

3. Prioritäten

Priorität Bedeutung Beispiel
P0 Launch unmöglich. Build kaputt, Kontaktweg falsch, Seite nicht erreichbar.
P1 Vor Launch beheben. Mobile CTA abgeschnitten, falsche Telefonnummer, Datenschutz fehlt.
P2 Vor Launch sinnvoll, aber kein harter Blocker. Ein Abschnitt wirkt noch zu dicht, Headline könnte besser brechen.
P3 Nach Launch möglich. Feinschliff an einzelnen Formulierungen oder kleinen Abständen.

4. Der Abschluss-Workflow

Phase 1: Technische Basis einfrieren

Erst sicherstellen, dass das Projekt technisch gesund ist.

  1. npm run lint
  2. npx tsc --noEmit
  3. npm run build
Bitte führe die technischen Basischecks aus: npm run lint, npx tsc --noEmit und npm run build. Berichte nur Fehler oder bestätige, dass die Basis sauber ist.

Phase 2: Seiteninventar und Screenshot-Set festlegen

Nicht zufällig prüfen. Lege vorher fest, welche Seiten und Viewports zählen.

Pflichtseiten:

  • /
  • /leistungen
  • /leistungen/unfallgutachten
  • /leistungen/autovermietung
  • /leistungen/autoservice
  • /leistungen/detailing
  • /leistungen/zulassungsservice
  • /leistungen/abschleppdienst-pannenhilfe
  • /kontakt
  • /ueber-uns
  • /impressum, /datenschutz, /agb

Pflicht-Viewports:

  • Mobile: 390x844, 430x932
  • Desktop: 1440x1100, 2048x996
Erstelle ein Seiteninventar für alle vorhandenen App-Routen und markiere die wichtigsten Launch-Seiten. Danach erstelle oder plane Screenshots für Mobile 390x844/430x932 und Desktop 1440x1100/2048x996.

Phase 3: Nur sammeln, nicht sofort fixen

Jetzt laufen die Prüfer. Ziel ist eine Finding-Liste, keine direkte Umsetzung.

  1. typography_checker
  2. mobile_visual_checker
  3. desktop_visual_checker
  4. content_consistency_reviewer
  5. conversion_reviewer
Nutze typography_checker, mobile_visual_checker, desktop_visual_checker, content_consistency_reviewer und conversion_reviewer. Sammle nur Findings, setze nichts um. Sortiere alles nach P0/P1/P2/P3.

Phase 4: Muster erkennen

Jetzt nicht einzelne Zeilen reparieren. Erst prüfen, ob es ein wiederkehrendes Muster gibt.

  • Sind Überschriften generell zu breit?
  • Fehlen measure-*-Klassen?
  • Sind Cards zu eng oder Texte zu lang?
  • Entstehen mobile Probleme durch ein gemeinsames Section-Pattern?
  • Werden CTA-Texte zu lang und brauchen button-text-wrap?
Analysiere die gesammelten Findings und gruppiere sie nach Ursache. Ich will keine Einzelkorrekturen, sondern Muster: Typografie-System, Section-Abstände, Card-Layout, CTA-Texte, Mobile-Header, Content-Länge.

Phase 5: Gebündelt umsetzen

Umsetzung nur blockweise. Nach jedem Block wird geprüft.

  1. Typografie-System und Textbreiten
  2. Mobile Layout/Overflow
  3. Desktop Layout/Weißraum
  4. Content-Konsistenz und CTA-Texte
  5. SEO, Legal/Trust, Performance
Setze nur die P1-Findings aus dem Bereich Typografie und Textbreiten um. Vorher nenne die Dateien, die du ändern wirst. Keine P2/P3-Findings anfassen.

Phase 6: Regression statt neue Gestaltungsrunde

Nach einem Fix-Block wird geprüft, ob der Block funktioniert und nichts kaputtging.

  1. typography_checker für betroffene Seiten
  2. mobile_visual_checker für betroffene Seiten
  3. desktop_visual_checker für betroffene Seiten
  4. a11y_checker, wenn UI betroffen ist
  5. quality_reviewer
Prüfe nur Regressionen für die gerade geänderten Seiten. Melde neue echte P0/P1-Probleme. Keine neuen P3-Feinschliff-Ideen sammeln.

5. Konkrete Regeln gegen den Zeilenumbruch-Kreislauf

  • Zeilenumbrüche nicht einzeln manuell korrigieren, wenn das Problem aus Breite, Schriftgröße oder Textlänge entsteht.
  • Erst measure-*, text-*-fluid und bessere Formulierung prüfen.
  • Manuelle Umbrüche nur für echte Hero- oder CTA-Ausnahmen einsetzen.
  • Wenn ein Problem auf drei Seiten ähnlich auftritt, ist es ein Systemproblem.
  • Wenn ein Problem nur auf einem Screenshot minimal unschön ist, aber nichts blockiert, ist es P2/P3.
  • Nach einem Fix nicht sofort neue ästhetische Ideen verfolgen; erst den geplanten Block abschließen.

6. Wann welcher Agent genutzt wird

Situation Agenten
Neue Komponente möglich codebase_explorer, danach bei UI a11y_checker, am Ende quality_reviewer
Zeilenumbrüche, Schrift, Textbreiten typography_checker, danach bei Screenshots mobile_visual_checker und desktop_visual_checker
Mobile wirkt kaputt mobile_visual_checker, danach bei UI-Fixes a11y_checker
Desktop wirkt nicht hochwertig desktop_visual_checker, danach premium_reviewer
Texte wirken uneinheitlich content_consistency_reviewer, danach typography_checker
Kontakt/Journey prüfen conversion_reviewer
Launch vorbereiten site-launch-audit, danach launch_reviewer

7. Prompts für deinen Abschluss

Startprompt für geordneten Abschluss

Nutze site-launch-audit. Ich will das Projekt strukturiert abschließen und nicht mehr in Einzelkorrekturen hängen bleiben. Prüfe zuerst technische Basis, dann sammle Findings für Mobile, Desktop, Typografie, Content, Conversion, SEO, Legal/Trust, Performance und Premium. Setze nichts um. Gib mir am Ende eine priorisierte P0/P1/P2/P3-Liste.

Prompt gegen Mikro-Fixes

Bitte keine einzelnen Zeilenumbrüche sofort korrigieren. Sammle alle typografischen und visuellen Findings zuerst, gruppiere sie nach Ursache und schlage dann maximal drei systemische Fix-Blöcke vor.

Typografie gezielt prüfen

Starte typography_checker für alle Hauptseiten. Prüfe §22: fluid type utilities, measure-Klassen, harte Zeilenumbrüche, Button-Wrapping und semantische Zeilenumbrüche. Berichte nur P1/P2, keine P3-Feinschliffe.

Mobile visuell prüfen

Starte mobile_visual_checker für /, /leistungen, alle Leistungsdetailseiten und /kontakt. Nutze 390x844 und 430x932. Prüfe Overflow, abgeschnittene Texte, CTA-Erreichbarkeit, Header/Nav und Touch-Ziele. Setze nichts um.

Desktop visuell prüfen

Starte desktop_visual_checker für /, /leistungen, alle Leistungsdetailseiten, /kontakt und /ueber-uns. Nutze 1440x1100 und 2048x996. Prüfe Zeilenumbrüche, Weißraum, Bildbeschnitt, visuelle Hierarchie und Premium-Eindruck.

Findings gruppieren

Fasse die Findings aus Typography, Mobile und Desktop zusammen. Gruppiere sie nach Ursache statt nach Seite. Priorisiere P0/P1/P2/P3. Markiere, welche Probleme systemisch gelöst werden sollten.

Nur P1 umsetzen

Setze nur die P1-Findings aus der gruppierten Liste um. Vorher nenne die betroffenen Dateien und den geplanten Fix-Block. Keine P2/P3-Punkte anfassen.

Regression prüfen

Prüfe nach den Änderungen nur Regressionen auf den betroffenen Seiten. Nutze typography_checker, mobile_visual_checker, desktop_visual_checker, a11y_checker und quality_reviewer. Melde nur neue P0/P1-Probleme.

Finaler Launch-Review

Starte launch_reviewer und fasse alle abgeschlossenen Prüfungen zusammen. Entscheide, ob die Website launch-freigebbar ist. Liste nur verbleibende P0/P1-Blocker und P2-Empfehlungen.

8. Praktischer Wochenplan

Block Ziel Agenten

9. Definition von fertig

  • Keine P0/P1-Blocker offen.
  • npm run lint erfolgreich.
  • npx tsc --noEmit erfolgreich.
  • npm run build erfolgreich.
  • Mobile-Hauptseiten in 390x844 und 430x932 geprüft.
  • Desktop-Hauptseiten in 1440x1100 und 2048x996 geprüft.
  • Telefon, WhatsApp und Kontaktwege sind korrekt.
  • Kontaktformular wird nicht als funktionierender Versand überverkauft, solange Backend fehlt.
  • Impressum, Datenschutz und Footer sind erreichbar und konsistent.
  • launch_reviewer meldet launch-freigebbar oder nur akzeptierte P2/P3-Restpunkte.
